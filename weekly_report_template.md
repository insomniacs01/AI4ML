## 三、项目整体情况

### 1. 项目当前定位

本项目的目标不是单纯实现一个自动调参工具，而是构建一个面向教学和实验场景的 AI4ML 低代码/零代码社区平台。平台希望支持用户通过自然语言描述任务并上传原始数据，由多智能体协作完成数据分析、特征处理、模型选择、训练、结果输出与基础部署。同时，平台还需要支持管理员治理、工作流复用与社区共享。

结合当前课程进度，小组将项目拆分为两个层次：

- 底层能力层：负责自动完成机器学习实验流程，包括数据分析、模型训练、代码生成、实验迭代和结果返回。
- 平台产品层：负责聊天交互、文件上传、任务管理、进度可视化、管理员治理、模型与数据集展示、人机协同等功能。

当前阶段重点不是一次性实现完整社区，而是先把“用户输入任务并上传数据，系统完成一次基础 AutoML 建模并返回结果”的最小闭环做出来。

### 2. 三类用户角色需求梳理

本周对项目文档中的需求进行了重新归纳，明确了三类主要用户：

#### （1）管理员

管理员主要关注平台治理和资源控制，核心需求包括：

- 管理用户权限与平台访问能力。
- 管理 LLM/API Token 与实验计算资源额度，防止资源滥用。
- 审核用户上传或公开分享的数据集、模型和工作流。
- 对数据中心和模型广场进行分类、打标签和内容维护。

#### （2）零基础业务用户

零基础用户主要关注“能否快速得到可用结果”，核心需求包括：

- 用自然语言描述任务目标。
- 上传 CSV 等基础数据文件。
- 看到系统执行过程和当前进度，而不是黑盒等待。
- 获得模型结果、评估指标和通俗易懂的分析报告。
- 在网页上进行简单测试，验证模型是否可用。

#### （3）有经验的 AI 开发者/数据科学家

高级用户主要关注“可控性、可修改性和可复用性”，核心需求包括：

- 在关键节点介入智能体流程，实现 Human-in-the-loop。
- 查看和修改系统生成的 Python 代码或 Pipeline。
- 导出工作流到本地环境继续开发。
- 分享优化后的工作流、提示词模板或模型产物供他人复用。

### 3. Alpha 版功能范围

为了保证课程项目在可控时间内形成可运行版本，本周对 Alpha 版功能做了明确收缩。Alpha 版只聚焦最小可行闭环，不追求完整社区能力。

Alpha 版计划支持的功能：

- 用户通过自然语言输入机器学习任务描述。
- 用户上传一份 CSV 表格数据。
- 系统自动识别任务类型并执行基础数据分析。
- 系统完成一次自动建模流程，优先支持表格数据的分类/回归任务。
- 系统输出模型性能结果、简单报告和可下载代码。
- 前端展示基础执行进度，例如数据分析、模型选择、训练完成等状态。

Alpha 版暂不纳入的功能：

- 图像、文本、时间序列、图数据等多模态任务。
- 完整的数据中心、模型广场与社区内容审核系统。
- 复杂的人机协同编辑器。
- 多用户协作和完善的权限体系。
- 一键部署完整线上推理服务。
- 高强度复杂动画与重交互前端效果。

### 4. 技术路线初步结论

结合助教建议和本周调研，小组不计划从零实现完整的多智能体 AutoML 内核，而是采用“开源项目二次开发”的方式推进。

当前初步结论如下：

- 优先选择成熟开源项目作为实验执行底座。
- 在其基础上自研任务管理、前端交互、进度可视化和角色权限等平台层功能。
- 所有二次开发部分将明确标注哪些能力来自开源项目，哪些部分为小组新增扩展。

## 四、本周执行情况

### 1. 需求分析与问题拆解

本周首先完成了现有需求文档的阅读和重组，对项目的“目标用户是谁、最核心的使用场景是什么、第一阶段必须做什么”进行了统一。

目前小组形成的判断是：

- 项目最核心的价值不在于重新造一个 AutoML 轮子，而在于把 AI 驱动的机器学习实验能力产品化、可视化和社区化。
- 如果一开始就同时追求多模态、多角色、社区、模型市场、人机协同、部署能力，项目范围会过大，难以在课程周期内完成。
- 因此第一阶段必须明确最小闭环，并围绕这个闭环倒推技术选型和分工。

### 2. 开源框架与论文调研结果

本周重点调研了当前“使用 AI 进行机器学习实验”的相关 GitHub 仓库与论文，筛选标准包括：

- 是否公开可获取。
- 是否具备机器学习实验自动化能力。
- 是否与自然语言交互、多智能体协作、实验迭代相关。
- 是否适合作为课程项目的二次开发基础。
- 是否具备较明确的许可证，便于后续合规使用。

调研后重点关注了以下几个方向：

| 项目/论文 | 类型 | 主要特点 | 是否适合作为底座 |
| --- | --- | --- | --- |
| AutoGluon Assistant / MLZero | GitHub + 论文 | 端到端多模态 ML 自动化，支持 WebUI/CLI/API，工程成熟度较高 | 是，优先级最高 |
| AIDE | GitHub + 论文 | 以树搜索方式迭代机器学习代码，过程可视化较强 | 可作为进度展示和搜索机制参考 |
| AutoML-Agent | GitHub + 论文 | 多智能体拆分明确，强调任务规划和多阶段验证 | 适合做架构参考，不建议直接做底座 |
| RD-Agent | GitHub | 更偏研发与研究助手，能力强但集成成本高 | 暂不作为 Alpha 首选 |
| FLAML / AutoGluon | GitHub | 传统 AutoML 能力成熟，可作为模型训练基础组件 | 可作为算法基础能力补充 |
| MLE-bench / ML-Dev-Bench / MLE-Dojo | GitHub | 偏评测基准与环境 | 不做底座，适合后续评测 |

#### （1）重点候选一：MLZero / AutoGluon Assistant

小组认为目前最适合作为项目主底座的方向是 AutoGluon Assistant 对应的 MLZero 路线。其优势在于：

- 更接近“用户输入需求 + 上传数据 + 自动完成建模”的产品形态。
- 已经具备相对成熟的工程接口，包含 WebUI、CLI、Python API。
- 对我们 Alpha 阶段所需的“表格数据自动建模”场景契合度较高。
- 许可证较清晰，便于后续在课程项目中进行二次开发说明。

如果后续技术验证顺利，团队计划优先参考这一方向，作为实验执行层的候选底座。

#### （2）重点候选二：AIDE

AIDE 的最大价值不在于直接作为完整平台使用，而在于它的两项能力非常值得借鉴：

- 将机器学习实验看作“代码空间中的搜索与迭代”问题。
- 对实验过程和搜索路径有较清晰的可视化表达。

这与我们平台希望展示“Agent Manager 正在拆解任务、Data Agent 正在分析数据、Model Agent 正在尝试模型”的需求高度吻合。因此，AIDE 更适合作为任务进度可视化和实验树展示的设计参考。

#### （3）重点候选三：AutoML-Agent

AutoML-Agent 的优势是角色拆分非常清晰，包含数据处理、模型决策、任务管理、操作执行等多个模块，并强调多阶段验证。这一点与我们的项目设想高度相似。

但从本周调研结果看，其更适合作为架构设计蓝本，而不是第一阶段直接复用的工程底座。原因主要包括：

- 工程成熟度仍偏研究型。
- 直接复用的产品化成本较高。
- 许可证和使用边界需要谨慎评估。

因此，小组决定将其作为“多智能体系统设计参考”，而非 Alpha 阶段的首选实现基础。

### 3. 当前技术路线判断

基于调研结果，本周形成的技术路线如下：

- 采用“现有开源 AutoML/Agent 系统 + 自研平台层”的实现思路。
- 第一阶段先支撑表格数据分类/回归场景。
- 优先打通前端输入、文件上传、任务创建、后端调用实验执行内核、返回结果与进度展示的完整链路。
- 复杂社区能力和高级人机协同能力放入后续迭代。

这一技术路线的优点在于：

- 能显著降低底层算法和实验编排的重复工作量。
- 有助于快速形成 Alpha 版最小闭环。
- 能将团队精力集中到更能体现课程项目特色的平台层扩展上。

### 4. 本周已识别的风险与问题

虽然整体技术方向已经基本清楚，但本周也识别出若干风险：

#### （1）项目范围过大风险

如果同时追求完整社区、复杂多角色交互、多模态任务和强人机协同，开发工作量会明显超过课程周期承受范围。

应对策略：

- 明确 Alpha 版只做表格数据闭环。
- 后续版本再逐步引入社区化和高级协同能力。

#### （2）开源项目集成复杂度风险

一些 Agent 型项目虽然能力先进，但依赖环境复杂、部署条件苛刻，可能导致“研究效果很好、工程接不起来”。

应对策略：

- 先对 1 个优先底座做小规模验证。
- 如果验证成本过高，及时切换到更稳妥的 AutoML 框架组合方案。

#### （3）团队分工边界不清风险

如果需求分析、技术调研、前后端开发和实验内核验证没有明确分工，容易出现重复劳动或关键模块无人负责。

应对策略：

- 以“产品/需求、前端、后端/Agent 集成、调研/测试”四条线明确责任人。
- 每周汇总进度并更新任务看板。
